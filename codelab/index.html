Introducción

⏱ 5 min

Codelab IA On-Device, Gemma 4 en Android

📅 Última actualización: mayo 2026 ⏱ Duración estimada: ~60 min

En este codelab construirás Validia, una app de inventario inteligente para Android que corre completamente offline. El usuario pega texto del proveedor en lenguaje natural (ej. "12 Coca-Cola y 6 Ala detergente") y Gemma 4 E2B extrae, clasifica y estructura los productos como JSON — sin internet, sin servidor, sin API key.

📝 Input: "12 Cokes, 4 Taragüi 500g, 2 Ala detergente"
🤖 Gemma 4 E2B → JSON
📦 Output: [{qty:12, Coca-Cola, Beverages}, ...]
🔒 100% local · 0 requests de red

El problema: IA en Android sin servidor

⏱ 5 min

El caso de uso

Imagina una pequeña tienda o depósito donde el proveedor entrega un texto informal: "Recibí 24 Coca-Cola 600ml, 6 bolsas de Lay's y 4 Ala detergente". El objetivo es convertir ese texto desestructurado en un inventario ordenado — automáticamente, sin conexión a internet.

La solución obvia sería llamar a una API como Gemini o GPT-4. Pero eso introduce:

  • Dependencia de red (sin internet = sin funcionalidad)
  • Latencia de round-trip (0.5–3 seg por consulta)
  • Costos por token en producción
  • Datos del negocio enviados a servidores externos (privacidad)

La alternativa es correr el modelo localmente en el dispositivo. Esto es posible hoy gracias a los modelos edge como Gemma 4 E2B (2 mil millones de parámetros, optimizado para inferencia móvil).

¿Por qué LiteRT-LM y no MediaPipe?

El formato .litertlm es el formato nativo de Google AI Edge. Al inicio del proyecto se evaluó MediaPipe Tasks GenAI, pero presentaba varios problemas con este formato de modelo:

Aspecto MediaPipe 0.10.33 + .litertlm LiteRT-LM 0.10.0 ✅
Chat template Manual (start_of_turn, end_of_turn) Automático via ConversationConfig
Token EOS No reconocido → early-stop manual Reconocido nativamente
API de streaming Callbacks + futures Kotlin Flow<Message> nativo
GPU / NPU Limitado Backend.GPU() / Backend.NPU()
Gestión de recursos Compleja (close en future.addListener) Simple: use {} / AutoCloseable
Decisión de arquitectura Usar com.google.ai.edge.litertlm:litertlm-android:0.10.0 simplifica enormemente el código: sin templates manuales, sin early-stop, sin manejo de futures. El SDK maneja el formato Gemma IT internamente.

Descarga el modelo Gemma 4 E2B

⏱ 10 min (depende de tu conexión)

¿Qué es Gemma 4 E2B?

Gemma 4 E2B es una variante de Gemma 4 con 2 mil millones de parámetros ("E2B" = Edge 2B), optimizada específicamente para inferencia en dispositivos móviles y edge. El formato .litertlm es el formato binario nativo del runtime LiteRT de Google.

  • Tamaño del archivo: ~2.4 GB
  • Formato: .litertlm (cuantizado, optimizado para ARM)
  • Variante: Instruction-tuned (-it): entrenado para seguir instrucciones
Acepta los términos de uso Debes aceptar la licencia de Gemma en Hugging Face antes de poder descargar el modelo. Necesitas una cuenta de Hugging Face.

Pasos para descargar

  1. Ve a https://huggingface.co/litert-community/Gemma4-4B-IT
  2. Inicia sesión o crea una cuenta en Hugging Face
  3. Acepta los términos de uso del modelo Gemma
  4. Descarga el archivo gemma-4-E2B-it.litertlm (busca la pestaña Files and versions)

pip install huggingface_hub

¿Por qué en assets y no descargado en runtime?

El modelo se empaqueta en assets/ para garantizar disponibilidad completamente offline desde el primer uso. No depende de servidores externos, play store feature delivery ni conexión de red en ningún momento. La app lo copia a filesDir en el primer inicio (paso 6).

Copia el modelo al almacenamiento interno

⏱ 5 min

¿Por qué necesitamos copiar el modelo?

El SDK de LiteRT-LM (Engine) requiere una ruta absoluta en el sistema de archivos del dispositivo. Los assets de Android no tienen una ruta física accesible directamente — solo se pueden abrir como InputStream.

La solución es copiar el modelo desde assets/ al directorio privado de la app (context.filesDir) la primera vez que se lanza. Las siguientes aperturas detectan que el archivo ya existe y se saltan la copia.

Crea el archivo data/util/ModelCopier.kt: 

package com.camachoyury.validia.data.util

import android.content.Context
import kotlinx.coroutines.Dispatchers
import kotlinx.coroutines.withContext
import java.io.File
import java.io.FileOutputStream

/**
 * Copia el modelo Gemma desde assets/ a filesDir.
 *
 * Engine de LiteRT-LM necesita una ruta absoluta en el filesystem.
 * Android assets solo se abren como InputStream, no tienen ruta física.
 *
 * Optimizaciones:
 * - Idempotente: si el archivo ya existe, no copia.
 * - Buffer de 8 KB: maneja archivos grandes (>1 GB) sin OOM.
 * - Ejecuta en Dispatchers.IO para no bloquear el main thread.
 */
object ModelCopier {

    private const val BUFFER_SIZE = 8 * 1024 // 8 KB

    /**
     * Copia el asset a filesDir si no existe ya.
     * @param assetName Nombre del archivo en assets/ (ej: "gemma-4-E2B-it.litertlm")
     * @return Ruta absoluta del archivo en filesDir.
     */
    suspend fun ensureModelReady(
        context: Context,
        assetName: String
    ): String = withContext(Dispatchers.IO) {
        val destFile = File(context.filesDir, assetName)

        if (!destFile.exists()) {
            context.assets.open(assetName).use { input ->
                FileOutputStream(destFile).use { output ->
                    val buffer = ByteArray(BUFFER_SIZE)
                    var bytesRead: Int
                    while (input.read(buffer).also { bytesRead = it } != -1) {
                        output.write(buffer, 0, bytesRead)
                    }
                    output.flush()
                }
            }
        }

        destFile.absolutePath
    }
}data/util/ModelCopier.kt

Puntos clave

  • destFile.exists(): la comprobación hace que la copia sea idempotente. El archivo de ~2.4 GB solo se copia una vez en toda la vida de la app.
  • Buffer de 8 KB: evita cargar el archivo completo en RAM. Crítico para archivos grandes.
  • withContext(Dispatchers.IO): la operación de disco se ejecuta en el thread pool de I/O, nunca en el main thread.
  • .use {}: cierra automáticamente los streams, incluso si hay una excepción.

Interfaz del motor de inferencia

⏱ 5 min

¿Por qué una interfaz?

En lugar de que el repositorio dependa directamente de la implementación de LiteRT, usamos una interfaz LocalInferenceEngine. Esto nos permite:

  • Intercambiar implementaciones (LiteRT, MediaPipe, stub de testing) sin cambiar el repositorio
  • Hacer unit tests con un fake del engine
  • Aislar el dominio de los detalles del SDK

Crea data/ai/LocalInferenceEngine.kt:

package com.camachoyury.validia.data.ai

/**
 * Contrato para el motor de inferencia on-device.
 *
 * Implementación activa:
 *  - [LiteRtInferenceEngine] → litertlm-android:0.10.0
 *
 * El engine maneja internamente:
 *  - El chat template del modelo (system instruction)
 *  - Tokens EOS (fin de generación nativo)
 *  - Streaming token a token via Flow<Message>
 *  - Aceleración GPU / NPU (configurable via Backend)
 */
interface LocalInferenceEngine {

    /**
     * Inicializa el engine cargando el modelo en memoria.
     * Operación costosa (~2-8 seg según dispositivo) — siempre en Dispatchers.IO.
     *
     * @param modelPath Nombre del asset (.litertlm) o ruta absoluta al archivo.
     */
    suspend fun initialize(modelPath: String)

    /**
     * Ejecuta inferencia local con el modelo cargado.
     *
     * El SDK maneja el chat template de Gemma IT internamente.
     * No es necesario construir manualmente <start_of_turn>user...
     *
     * @param systemPrompt Instrucciones del modelo (few-shot, reglas, formato).
     * @param userInput    Texto del usuario a procesar.
     * @param onToken      Callback invocado por cada chunk de token generado.
     * @return Texto generado por el modelo (se espera: JSON array).
     */
    suspend fun generate(
        systemPrompt: String,
        userInput: String,
        onToken: (String) -> Unit = {}
    ): String

    /** Libera recursos nativos. Llamar en ViewModel.onCleared(). */
    fun close()
}data/ai/LocalInferenceEngine.kt

Puntos clave del diseño

  • initialize y generate son suspend: ambas operaciones son potencialmente largas y deben correr en coroutines, nunca en el main thread.
  • onToken: (String) -> Unit: callback de streaming que el ViewModel usa para actualizar el contador de tokens en tiempo real en la UI.
  • close(): libera la RAM nativa del modelo. Es crítico llamarlo en ViewModel.onCleared() para evitar memory leaks.

LiteRtInferenceEngine: implementación con LiteRT-LM

⏱ 10 min

Esta es la clase central del proyecto. Implementa LocalInferenceEngine usando el SDK litertlm-android:0.10.0.

Crea data/ai/LiteRtInferenceEngine.kt:

1. Configuración y carga del modelo

class LiteRtInferenceEngine(
    private val context: Context
) : LocalInferenceEngine {

    private var engine: Engine? = null

    companion object {
        const val TAG = "Gemma4Debug"

        // temperature=0.1 + topK=1 = máxima determinismo para salidas JSON
        private const val TEMPERATURE = 0.1
        private const val TOP_P       = 0.9
        private const val TOP_K       = 1
        private const val SEED        = 0
    }

    override suspend fun initialize(modelPath: String) = withContext(Dispatchers.IO) {
        val physicalPath = if (modelPath.startsWith("/")) {
            modelPath // ya es ruta absoluta
        } else {
            // Es un nombre de asset → copiar a filesDir primero
            ModelCopier.ensureModelReady(context, modelPath)
        }

        val engineConfig = EngineConfig(
            modelPath = physicalPath,
            backend   = Backend.CPU(),   // CPU: máxima compatibilidad
            // Backend.GPU() → GPU vía OpenCL (Adreno, Mali)
            // Backend.NPU() → NPU vía vendor libs
            cacheDir  = context.cacheDir.path // acelera cargas futuras
        )

        val e = Engine(engineConfig)
        e.initialize()
        engine = e
    }data/ai/LiteRtInferenceEngine.kt

2. Inferencia con Flow nativo

    override suspend fun generate(
        systemPrompt: String,
        userInput: String,
        onToken: (String) -> Unit
    ): String = withContext(Dispatchers.IO) {
        val e = engine ?: throw IllegalStateException("Engine no inicializado.")

        // ConversationConfig inyecta el system instruction y parámetros de sampling.
        // El SDK aplica el chat template correcto para Gemma IT internamente.
        // No se necesita <start_of_turn>user...<end_of_turn><start_of_turn>model.
        val conversationConfig = ConversationConfig(
            systemInstruction = Contents.of(systemPrompt),
            samplerConfig = SamplerConfig(
                topK        = TOP_K,
                topP        = TOP_P,
                temperature = TEMPERATURE,
                seed        = SEED
            )
        )

        val accumulated = StringBuilder()

        // createConversation.use {} → cierra recursos automáticamente.
        // sendMessageAsync → Flow<Message>: cada emisión es un chunk de token.
        // El Flow termina cuando el modelo emite el token EOS (reconocido nativamente).
        // No se necesita cancelación manual ni detectar el ']' del JSON.
        e.createConversation(conversationConfig).use { conversation ->
            conversation
                .sendMessageAsync(userInput)
                .collect { message ->
                    val chunk = message.toString()
                    onToken(chunk)
                    accumulated.append(chunk)
                }
        }

        accumulated.toString()
    }

    override fun close() {
        engine?.close()
        engine = null
    }
}data/ai/LiteRtInferenceEngine.kt

Parámetros de sampling explicados

ParámetroValorRazón
temperature0.1Salida casi determinista → JSON válido
topK1Selecciona solo el token más probable (greedy)
topP0.9Requerido por la API; a efectos prácticos con topK=1 no tiene impacto
seed0Semilla aleatoria; 0 = aleatorio pero reproducible entre sesiones
Tip: Debug con Logcat Filtra el tag Gemma4Debug en Logcat para ver: tiempo de carga del modelo, backend en uso, tokens en streaming, respuesta completa y tiempo total de inferencia.
Habilitar GPU (opcional) Cambia Backend.CPU() por Backend.GPU() para usar la GPU del dispositivo via OpenCL. Disponible en la mayoría de dispositivos con Adreno o Mali. La inferencia puede ser 3–5× más rápida.

Repositorio de inventario

⏱ 10 min

El repositorio es el corazón de la lógica de negocio. Se encarga de:

  1. Definir el system prompt con few-shot examples
  2. Llamar al engine con el prompt y el texto del usuario
  3. Extraer el JSON de la respuesta del modelo
  4. Parsear el JSON a objetos InventoryItem

Crea data/litert/LocalInventoryRepository.kt:

1. El System Prompt (Few-Shot)

private val systemInstructions = """
    You are an inventory assistant for stores and warehouses.
    Your only job is to extract products from a text and return them as JSON.

    STRICT RULES:
    - Return ONLY a valid JSON array. No extra text, no markdown.
    - Each object: "quantity" (Int), "productName" (String), "category" (String).
    - "category" MUST be one of: "Beverages", "Food" or "Cleaning".
    - If you cannot classify a product, default to "Food".

    EXAMPLES:

    Input: "got 24 Coca-Cola 20oz and 6 bags of Lay's chips"
    Output: [{"quantity":24,"productName":"Coca-Cola 20oz","category":"Beverages"},
             {"quantity":6,"productName":"Lay's Chips","category":"Food"}]

    Input: "order 4 Tide detergent and 3 bottles of vegetable oil"
    Output: [{"quantity":4,"productName":"Tide Detergent","category":"Cleaning"},
             {"quantity":3,"productName":"Vegetable Oil","category":"Food"}]

    Now process the following text:
""".trimIndent()
¿Por qué few-shot? Los modelos instruction-tuned responden mejor cuando ven ejemplos concretos de la transformación esperada. Los dos ejemplos del prompt enseñan al modelo exactamente el formato JSON requerido y cómo clasificar categorías.

2. Inicialización y cierre

class LocalInventoryRepository(
    private val engine: LocalInferenceEngine
) {
    private val json = Json {
        ignoreUnknownKeys = true
        isLenient = true   // tolera pequeñas imperfecciones del JSON generado
    }

    suspend fun initializeModel(modelPath: String) =
        withContext(Dispatchers.IO) { engine.initialize(modelPath) }

    fun closeModel() = engine.close()
}

3. Inferencia y parseo

    suspend fun parseInventoryText(
        rawText: String,
        onToken: (String) -> Unit = {}
    ): List<InventoryItem> = withContext(Dispatchers.IO) {
        require(rawText.isNotBlank()) { "El texto no puede estar vacío." }

        val rawResponse = engine.generate(
            systemPrompt = systemInstructions,
            userInput    = rawText,
            onToken      = onToken
        )

        val cleanJson = extractCleanJsonArray(rawResponse)
        json.decodeFromString<List<InventoryItem>>(cleanJson)
    }

    /**
     * Extrae el primer bloque JSON array de la respuesta del modelo.
     * Encuentra el primer '[' y el último ']', ignorando texto extra
     * que el modelo pudiera agregar a pesar del prompt estricto.
     */
    private fun extractCleanJsonArray(rawResponse: String): String {
        val start = rawResponse.indexOf('[')
        val end   = rawResponse.lastIndexOf(']')

        if (start == -1 || end == -1 || end <= start) {
            throw IllegalStateException(
                "No se encontró JSON array en la respuesta: \"${rawResponse.take(200)}\""
            )
        }

        return rawResponse.substring(start, end + 1)
    }data/litert/LocalInventoryRepository.kt

¿Por qué extractCleanJsonArray?

Los LLMs, incluso con prompts muy estrictos, a veces agregan texto introductorio antes o después del JSON (ej: "Sure! Here's the result:"). Esta función es una red de seguridad que recorta el array JSON limpio de cualquier texto extra.

ViewModel

⏱ 5 min

El InventoryViewModel orquesta el flujo completo: carga del modelo, inferencia, actualización de estado y limpieza de recursos.

Crea presentation/inventory/InventoryViewModel.kt:

class InventoryViewModel(
    private val repository: LocalInventoryRepository,
    private val modelPath: String
) : ViewModel() {

    private val _uiState    = MutableStateFlow<InventoryUiState>(InventoryUiState.Idle)
    val uiState: StateFlow<InventoryUiState> = _uiState.asStateFlow()

    private val _rawText = MutableStateFlow("")
    val rawText: StateFlow<String> = _rawText.asStateFlow()

    private val _debugState = MutableStateFlow<DebugState?>(null)
    val debugState: StateFlow<DebugState?> = _debugState.asStateFlow()

    init { initializeModel() }  // carga el modelo al crear el ViewModel

    // ── Carga del engine LiteRT ───────────────────────────────────────
    private fun initializeModel() {
        viewModelScope.launch {
            _uiState.value = InventoryUiState.InitializingModel
            try {
                repository.initializeModel(modelPath)
                _uiState.value = InventoryUiState.Ready
            } catch (e: Exception) {
                _uiState.value = InventoryUiState.Error(e.message ?: "Error al cargar el modelo")
            }
        }
    }

    fun retryInitialization() = initializeModel()

    // ── Inferencia on-device ──────────────────────────────────────────
    fun processText(rawText: String) {
        viewModelScope.launch {
            _uiState.value = InventoryUiState.Processing
            val startMs = SystemClock.elapsedRealtime()
            _debugState.value = DebugState(isRunning = true)

            try {
                val items = repository.parseInventoryText(rawText) { _ ->
                    // Actualiza el debug panel con cada token recibido
                    val current = _debugState.value ?: DebugState()
                    _debugState.value = current.copy(
                        tokenCount = current.tokenCount + 1,
                        elapsedMs  = SystemClock.elapsedRealtime() - startMs
                    )
                }

                val totalMs = SystemClock.elapsedRealtime() - startMs
                _debugState.value = _debugState.value?.copy(
                    isRunning = false, isCompleted = true, elapsedMs = totalMs
                )
                _uiState.value = InventoryUiState.Success(items)

            } catch (e: Exception) {
                _debugState.value = _debugState.value?.copy(isRunning = false)
                _uiState.value = InventoryUiState.Error(e.message ?: "Error en inferencia")
            }
        }
    }

    fun onTextChanged(newText: String) { _rawText.value = newText }

    fun reset() {
        _rawText.value = ""
        _uiState.value = InventoryUiState.Ready
        _debugState.value = null
    }

    // ── Limpieza de recursos nativos ──────────────────────────────────
    override fun onCleared() {
        super.onCleared()
        repository.closeModel()  // libera la RAM del Engine nativo
    }

    // ── Factory para wiring manual (sin Hilt) ────────────────────────
    class Factory(
        private val repository: LocalInventoryRepository,
        private val modelPath: String
    ) : ViewModelProvider.Factory {
        @Suppress("UNCHECKED_CAST")
        override fun <T : ViewModel> create(modelClass: Class<T>): T =
            InventoryViewModel(repository, modelPath) as T
    }
}presentation/inventory/InventoryViewModel.kt
onCleared() es crítico El Engine de LiteRT-LM reserva varios GB de RAM nativa. Si no se llama a closeModel() en onCleared(), esa memoria no se libera cuando el usuario abandona la pantalla.

UI con Jetpack Compose

⏱ 10 min

La pantalla principal se llama SmartInventoryScreen. Sigue el patrón State Hoisting: el composable recibe estado y lambdas, nunca el ViewModel directamente.

Estructura de la pantalla

SmartInventoryRoute (lee StateFlows del ViewModel)
  └─ SmartInventoryScreen (estado puro, sin ViewModel)
      ├─ Header (siempre visible)
      ├─ ModelLoadingSection (InitializingModel | Idle)
      ├─ InputSection (Ready | Processing)
      ├─ InferenceLoadingSection (Processing)
      ├─ LazyColumn + InventoryItemCard (Success)
      ├─ ErrorSection (Error)
      └─ DebugPanel (visible cuando debugState != null)

Route — punto de contacto con el ViewModel

@Composable
fun SmartInventoryRoute(
    viewModel: InventoryViewModel,
    modifier: Modifier = Modifier
) {
    val rawText    by viewModel.rawText.collectAsState()
    val uiState    by viewModel.uiState.collectAsState()
    val debugState by viewModel.debugState.collectAsState()

    SmartInventoryScreen(
        rawText          = rawText,
        uiState          = uiState,
        debugState       = debugState,
        onRawTextChange  = viewModel::onTextChanged,
        onProcessClick   = viewModel::processText,
        onResetClick     = viewModel::reset,
        onRetryClick     = viewModel::retryInitialization,
        modifier         = modifier
    )
}

Screen — lógica de estado con when

@Composable
fun SmartInventoryScreen(...) {
    Column(modifier = modifier.fillMaxSize().padding(horizontal = 16.dp)) {
        // Header siempre visible
        Text(text = "Smart Inventory Assistant", ...)
        Text(text = "Gemma 4 E2B · LiteRT · 100% offline", ...)

        when (uiState) {
            is InventoryUiState.InitializingModel,
            is InventoryUiState.Idle      -> ModelLoadingSection()
            is InventoryUiState.Ready     -> InputSection(...)
            is InventoryUiState.Processing -> {
                InputSection(isEnabled = false, ...)
                InferenceLoadingSection()
            }
            is InventoryUiState.Success   -> {
                LazyColumn { items(uiState.items) { InventoryItemCard(it) } }
            }
            is InventoryUiState.Error     -> ErrorSection(...)
        }

        // Debug panel — siempre visible cuando hay datos
        if (debugState != null) DebugPanel(debugState = debugState!!)
    }
}

CategoryChip — chip con color semántico por categoría

@Composable
private fun CategoryChip(category: String) {
    val (bg, fg) = when (category) {
        "Beverages" -> Color(0xFFBBDEFB) to Color(0xFF0D47A1)  // azul
        "Food"      -> Color(0xFFC8E6C9) to Color(0xFF1B5E20)  // verde
        "Cleaning"  -> Color(0xFFFFF9C4) to Color(0xFFF57F17)  // amarillo
        else        -> Color(0xFFE0E0E0) to Color(0xFF424242)  // gris
    }
    Surface(color = bg, shape = RoundedCornerShape(50.dp)) {
        Text(text = category, modifier = Modifier.padding(10.dp, 4.dp), color = fg)
    }
}

Conecta todo en MainActivity

⏱ 3 min

MainActivity es el punto de ensamblaje del grafo de dependencias manual. Sin Hilt, cableamos el engine → repositorio → ViewModel directamente aquí.

Edita MainActivity.kt:

class MainActivity : ComponentActivity() {

    /**
     * Nombre del archivo de modelo en assets/.
     * LiteRtInferenceEngine lo copiará a filesDir en el primer inicio.
     * Ubicación: app/src/main/assets/gemma-4-E2B-it.litertlm
     */
    private val modelAssetName = "gemma-4-E2B-it.litertlm"

    /**
     * Grafo de dependencias manual (sin Hilt):
     *   LiteRtInferenceEngine
     *     ↓ implementa LocalInferenceEngine
     *   LocalInventoryRepository(engine)
     *     ↓
     *   InventoryViewModel(repository, modelPath)
     */
    private val viewModel: InventoryViewModel by viewModels {
        InventoryViewModel.Factory(
            repository = LocalInventoryRepository(
                engine = LiteRtInferenceEngine(applicationContext)
            ),
            modelPath = modelAssetName
        )
    }

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        enableEdgeToEdge()
        setContent {
            ValidiaTheme {
                Scaffold(modifier = Modifier.fillMaxSize()) { innerPadding ->
                    SmartInventoryRoute(
                        viewModel = viewModel,
                        modifier  = Modifier.padding(innerPadding)
                    )
                }
            }
        }
    }
}MainActivity.kt
¿Sin Hilt? El grafo de dependencias de esta app es simple y lineal. Usar Hilt agregaría complejidad sin beneficio real. Para proyectos más grandes con múltiples pantallas y repositorios compartidos, sí se recomendaría Hilt o Koin.

¡Felicidades!

Construiste una app Android con IA generativa 100% offline usando Gemma 4 E2B y LiteRT-LM.

¿Qué construiste?

  • Motor de inferencia on-device usando litertlm-android:0.10.0 con el formato .litertlm nativo
  • Few-shot prompting para extraer JSON estructurado de texto libre en lenguaje natural
  • Arquitectura limpia con separación de capas: engine → repositorio → ViewModel → UI
  • Streaming de tokens via Flow<Message> para actualizar la UI en tiempo real
  • UI reactiva con Jetpack Compose y gestión completa de estados (carga del modelo, inferencia, resultados, errores)
